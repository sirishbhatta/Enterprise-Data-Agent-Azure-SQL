② Every File — What It Is and What It Does

Click any card to understand exactly what that file is responsible for.

FILE 1
app.py
Entry Point

The Front Door

The only file Streamlit executes directly. It sets the page title/icon, draws the sidebar branding and live database pills, then hands off to whichever page the user clicked. It knows about all other pages but does minimal work itself.

Run it with: streamlit run app.py --server.port 8502

FILE 2
bi_dashboard.py
Core AI Logic

The AI Brain

The largest and most complex file. Contains the entire agent pipeline: vector memory search, query rewriting, AI routing supervisor, multi-model SQL generation with automatic fallback, federated query merging, result display, and thumbs-up feedback loop.

Functions: rewrite_query() → supervisor_routing() → agent_execution()

FILE 3
db_connector.py
Database Engine

The Universal Connector

Reads db_registry.yaml and builds SQLAlchemy connections to any database type. All database activity in the app goes through here — nobody opens their own connection directly. This is the "one source of truth" for all DB access.

Key functions: build_engine(), execute_query(), discover_schema(), test_connection()

FILE 4
db_registry.yaml
Configuration

The Database Phonebook

A plain-text config file (YAML format = indented key: value pairs). Lists every database: its type, hostname, username, which .env variable holds the password, and the domain label the AI uses for routing. Add new databases here. No Python needed.

⚠️ Never put the actual password here — put the variable name (e.g. AZURE_SQL_PASSWORD) and store the value in .env.

FILE 5
.env
Secrets

Your Secrets File

Stores passwords and API keys as KEY=value lines. load_dotenv() in Python loads it into memory. os.getenv("KEY") retrieves a value. This file must never be shared — it's the vault. If you ever use Git, add .env to your .gitignore file.

FILE 6
schema_cache.json
Auto-Generated

The AI's Table Knowledge

Created automatically when you click "Discover Schema" in Connection Manager. Stores every table name, column name, data type, primary key, and foreign key for every active database. The AI reads this to write accurate SQL. Re-run Discover Schema after any table changes.

FILE 7.1
pages/1_Connection_Manager.py
Settings UI

Database Admin Panel

The settings page. Lets you: test live connections, discover and browse schemas, add new databases via form, enable/disable connections. The pages/ folder is a Streamlit convention — files here automatically become navigable pages.

FILE 8
requirements.txt
Dependencies

Package List

Lists all Python libraries the app needs. Run pip install -r requirements.txt once after a fresh install to get everything. Commented-out lines show optional packages for database types not yet activated.

FILE 9

Windows One-Click Start

Double-click this to start everything: opens Streamlit in one window, waits 5 seconds, then starts ngrok tunnels so your dashboard is reachable from the internet via https://sirish.ngrok.app.

④ Vector Memory — How It Really Works

This is one of the most powerful (and least obvious) features of your app. Here's a complete plain-English explanation.

What is a "vector" / "embedding"?

When the app needs to "remember" a question, it doesn't store the words — it converts the question into a list of 768 numbers. This list is called a vector or embedding. Each number captures a tiny piece of the question's meaning.

📍 The GPS analogy

A GPS coordinate (latitude, longitude) is a pair of numbers that uniquely identifies a location on Earth. Two nearby places have similar coordinates. An embedding is the same idea — but for meaning. Two questions that mean the same thing will have similar coordinates in a 768-dimensional space, even if they use completely different words.

🎨 The color analogy

The color "red" can be described as (255, 0, 0) in RGB — three numbers. "Dark red" is (139, 0, 0). They're different numbers but obviously similar. A question embedding works the same way — "top providers by billing" and "leading providers by total amount" will produce very similar 768-number vectors because they mean the same thing.

# This is what happens when you type "average salary by department":
vector = ollama.embeddings(model="nomic-embed-text", prompt="average salary by department")
# vector["embedding"] is now a list of 768 floats, e.g.:
# [0.0234, -0.1823, 0.7741, 0.0012, -0.3401, 0.9012, ...]  ← 768 numbers total
# This list IS the "meaning" of your question, captured mathematically

How similarity is measured — Cosine Distance

To find questions in memory that are similar to a new one, the app computes the cosine distance between two vectors. A distance of 0 means identical meaning; distance of 1 means completely unrelated. The app converts distance to a percentage: similarity = (1 - distance) × 100.

Example similarity scores against "average salary by department"

"What is the mean salary grouped by department?"94%
"Show me salaries for each team"81%
"Who are the top 3 providers by billed amount?"42%
"What is the weather in New York?"8%

The app triggers a memory hint at ≥65%. It skips saving a duplicate at ≥95%.

# SQL Server 2025 stores and searches vectors natively:
# The ai_query_cache table looks like this:
#   user_question      VARCHAR  — the original question text
#   sql_code           VARCHAR  — the SQL that worked
#   question_embedding VECTOR(768) — the 768 numbers representing the meaning

# Search: find the most similar past question
search_query = """
    SELECT TOP 1
        user_question, sql_code,
        VECTOR_DISTANCE('cosine', question_embedding, CAST(? AS VECTOR(768))) AS distance
    FROM ai_query_cache
    ORDER BY distance ASC;
"""
# ORDER BY distance ASC = smallest distance first = most similar question first

📜 The Evolution: From pgvector → SQL Server 2025

You asked about this — yes, the vector memory system changed. Here's the full history:

Era 1 — Early Days

No Memory

Every question was treated as brand new. The AI had to figure out the SQL from scratch every time. Fast but couldn't learn from past successes.

→
Era 2 — query_memory.json

JSON File Memory

Past queries saved to a JSON file. Simple text matching only — "average salary" would NOT match "mean pay". No semantic understanding. This is why all those entries had domain "UNKNOWN" — the domain tracking wasn't built yet.

→
Era 3 — pgvector (PostgreSQL)

pgvector Extension

You installed the pgvector extension in PostgreSQL and manually created the table and index. Required: CREATE EXTENSION vector; and running setup SQL. Worked but needed a separate PostgreSQL step just for memory.

→
Era 4 — NOW ✅

SQL Server 2025 Native Vectors

SQL Server 2025 added VECTOR(768) as a native column type and VECTOR_DISTANCE() as a built-in function. No extension to install. Your existing HealthcareClaims database now holds the AI memory in the ai_query_cache table.

✅ What this means for you: You no longer need to maintain a separate PostgreSQL setup just for vector memory. SQL Server 2025 handles it natively. The ai_query_cache table in your HealthcareClaims database IS your memory bank.
📋 The table you created manually in SQL Server: 
CREATE TABLE ai_query_cache (
    id                 INT IDENTITY PRIMARY KEY,
    user_question      NVARCHAR(MAX),
    sql_code           NVARCHAR(MAX),
    question_embedding VECTOR(768),   -- requires SQL Server 2025
    created_at         DATETIME DEFAULT GETDATE()
);

⑥ New Features Added

Six improvements implemented in the second session — all with educational notes.

① Row Limit Guardrail — db_connector.py

Every query now gets a hard cap of 500 rows automatically injected before it reaches the database, unless the query already has a TOP or LIMIT clause. This prevents a runaway SELECT * FROM Claims from returning 500,000 rows and crashing the browser.

DatabaseHow the limit is injectedExample
SQL Server / Azure SQLAdds TOP N after the first SELECTSELECT TOP 500 * FROM Claims
PostgreSQL / MySQL / SQLiteAppends LIMIT N at the endSELECT * FROM hr_employees LIMIT 500
OracleWraps in subquery with ROWNUM <= NSELECT * FROM (...) WHERE ROWNUM <= 500

The function returns a was_limited flag. When True, the UI shows a yellow warning: "Results capped at 500 rows — add a WHERE clause to narrow down." The limit can be changed by editing the MAX_ROWS = 500 constant in bi_dashboard.py. Scheduled reports use a higher cap of 5,000 rows since they're meant for full exports.

② FEDERATED Domain Routing is now Dynamic — bi_dashboard.py

Previously the federated query logic was hardcoded to only work with HR + CLAIMS. Now it reads from a FEDERATED_JOIN_CONFIGS dictionary. Adding a new database pair is just adding one entry to that dict — no other code changes needed.

The AI is now given query keys named after the actual domain (HR_query, CLAIMS_query) instead of the vague postgres_query/sql_server_query. This means the AI knows exactly which database each query goes to.

③ Azure SQL FEDERATED Support — bi_dashboard.py

Added frozenset(["HR", "AZURE_CLAIMS"]) to FEDERATED_JOIN_CONFIGS. Now you can ask questions like "Show HR employees with their Azure Claims history" and the federated engine will query PostgreSQL (HR) + Azure SQL (Claims) and merge them in Python — exactly the same way HR + local Claims works.

④ Row Count Warning — bi_dashboard.py

When results were automatically capped by the row limit, a yellow st.warning() box appears above the download button: "⚠️ Results capped at 500 rows. Add a WHERE clause to narrow down." This uses the was_limited flag returned by db_connector.execute_query(). The flag is also stored in the message metadata so the warning still shows when scrolling back through chat history.

⑤ Download as Excel — bi_dashboard.py

Every query result now has a 📥 Download as Excel button. Clicking it converts the DataFrame to an Excel file in memory (using BytesIO — no temp file written to disk) and serves it as a file download. The filename includes the timestamp so multiple downloads don't overwrite each other. Requires the openpyxl package (added to requirements.txt).

⑥ Scheduled Reports — pages/2_Scheduled_Reports.py

A brand new third page in the Settings section. Lets you:

  • Save any SQL query as a named report with a description, database domain, and run frequency
  • Run any report on demand with a ▶️ Run Now button
  • Download the result as Excel directly from the page
  • Run all saved reports at once with ▶️ Run All Reports
  • Generate a run_all_reports.bat + run_all_reports.py for Windows Task Scheduler

Form fix: st.form() for atomic widget submission

The Add New Report form is wrapped in st.form() instead of using plain buttons. This ensures all form inputs (name, description, domain, SQL, frequency, max_rows) are captured together in one atomic submission. Without this, Streamlit's button rerun could lose widget values from conditional form blocks. The fix uses st.form_submit_button() instead of st.button() — this guarantees the form values persist.

Run Frequency field for scheduling

Each report now stores a Run Frequency choice: "On Demand", "Daily", "Weekly (Mon)", or "Monthly (1st)". This is saved with the report in saved_reports.json and later used when generating the Windows Task Scheduler batch file. When you click ⚙️ Generate Batch File, the Task Scheduler section automatically reads all frequencies from your saved reports and summarizes them (e.g., "Daily, Weekly (Mon)") — you don't need to pick one globally anymore.

Reports are stored in saved_reports.json. Output Excel files go to reports_output/. The Task Scheduler setup walks you through adding the batch file to Windows — no extra tools needed.
