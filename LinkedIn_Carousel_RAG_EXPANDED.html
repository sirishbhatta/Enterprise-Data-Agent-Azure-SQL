Memory Search: Cosine Distance

4/9

How the system finds similar past questions: converts both to 768-dimensional vectors, measures angle (cosine distance). Result: 0 = identical, 1 = completely different.

Example similarity scores against "average salary by department":

"What is the mean salary grouped by department?" 94%
"Show me salaries for each team" 81%
"Who are the top 3 providers by billed amount?" 42%
"What is the weather in New York?" 8%
Thresholds: Memory hint triggers at ≥65% | Skip duplicate save at ≥95%

Evolution: pgvector → SQL Server 2025

5/9

The vector memory system evolved over 12 weeks:

Week 1
No Memory
Every question treated as brand new. AI regenerates SQL from scratch every time.
Week 4
JSON File Memory
Past queries saved to JSON. Simple text matching only — "average salary" ≠ "mean pay". No semantic understanding.
Week 8
pgvector (PostgreSQL)
Installed pgvector extension. Manual table + index setup. Worked but required separate PostgreSQL step.
Week 12 ✅
SQL Server 2025 Native Vectors
VECTOR(768) is now a native data type. No extension needed. HealthcareClaims database now holds AI memory in ai_query_cache table.

PATH B: Rewrite | PATH C: Memory Hit

6/9

PATH B: Follow-Up (Rewrite)

Scenario: User asks "what about just Sales?" after previous question.

Solution: AI detects follow-up, expands context. "What about just Sales?" → "What is average salary for Sales department?"

Why: Prevents AI from getting confused by out-of-context questions.

PATH C: Memory Hit

Scenario: User asked "top 3 providers by billing" before and gave 👍. Now asks "leading providers by revenue?"

Solution: Vector search finds 87% match. Past SQL injected as hint.

Result: Faster, more accurate. Badge shows "Memory Hit 87%"

PATH D: Cascade | PATH E: Federated

7/9

PATH D: Model Cascade

Problem: Local AI (gemma3:12b) generates bad SQL with syntax error.

Solution: Auto-escalates to next tier in cascade list.

Result: Tries gemini-2.5-flash → succeeds. Badge: "After 1 retry (gemini)"

Why: System never crashes. Always finds a working model.

PATH E: Federated Query

Problem: Question needs HR data (PostgreSQL) + Claims data (SQL Server cloud).

Solution: Generate 2 queries, run parallel, merge in Python (pandas).

Result: Single unified table with data from both databases.

Why: SQL can't natively join across different servers.

The Complete Picture

9/9

A RAG Agent with 5 Intelligent Paths

PATH A: Simple questions → direct execution (most common)

PATH B: Follow-ups → AI expands context from history

PATH C: Familiar questions → vector memory finds & reuses past SQL (faster)

PATH D: Model fails → cascade escalates to next tier (resilience)

PATH E: Multi-database → federated join in Python (cross-server queries)

Result: A system that learns, remembers, routes intelligently, scales across databases, and never crashes.
